# 转账链规则文档

> 本文档记录不同链的转账相关规则，包括最小转账金额、账户最低余额、Gas 规则、链特殊机制等。
> 生成转账测试用例时，必须参考本文档中的规则。

---

## 🧱 链专项转账用例标准结构（强制）

> 适用于 `docs/qa/testcases/cases/transfer/<chain>转账-*.md` 各链转账用例。

### 基本组成

链专项转账用例**只包含两类内容**：

1. **主流程参数化**：按 CSV / 用户提供的数据点（币种 × 金额）逐行覆盖完整流程（发送 → 选币 → 输入地址 → 输入金额 → 切法币 → 预览 → 确认 → 历史 → 区块浏览器对账）
2. **边界值用例**：基于该链特性的边界场景（如 EIP-1559 进阶设置 / Tron 资源不足烧 TRX / Polkadot Max 计算）

可选追加：性能与体验（loading / 重复进入 / 多链切换）。

### 转账双方账户固定（强制）

链专项转账用例的接收方**统一固定为「账户1 ↔ 账户2 互转」**，余额不足时自动反向（与 `helpers/runtime-config.mjs` 的 `resolveTransferDirection()` 行为对齐）。

- 不在场景列里给接收方分配「钱包内 / 地址簿 / 首次转账 / 链首次转账」等多种类型
- 不为了"覆盖地址识别 UI"而在不同行换接收方类型 — 这类通用 UI 行为不属于链用例职责
- 钱包账户标签 / 地址簿标签 / 首次转账提示等通用地址识别行为：归通用文档或 `wallet-rules.md`，链用例不重复

### 严禁单独建章节

| 禁止类型 | 应写在哪 |
| --- | --- |
| 链特殊机制描述（免 Gas 规则 / 资源模型 / ED 规则等） | **本规则文档**对应链章节 |
| 状态动态切换（处理中 → 成功 / 失败展示） | 通用历史记录行为，不写 |
| 产品体验建议 | 默认不写；如需则提交前自动删除（见 `qa-rules.md §9.5`） |
| 新地址 / 租金 / ED 边界用例 | `docs/qa/testcases/cases/wallet/2026-04-09_Wallet-发送-转账规则&租金.md` |
| 地址格式校验（SS58 / Base58 / 0x / 校验和） | 输入框通用校验逻辑覆盖，不写 |
| 余额为 0 前置拦截 | 同上租金用例 §0 |
| 接收方地址类型多样化（地址簿 / 首次转账 / 链首次转账） | 通用 UI 行为，链用例统一用「账户1 ↔ 账户2」 |

### 头部声明（强制）

链用例文档头部用 blockquote 引用通用文档，不重复正文：

```markdown
> 边界：**新地址 / 主币余额 = 0 前置拦截**等通用边界见 `docs/qa/testcases/cases/wallet/2026-04-09_Wallet-发送-转账规则&租金.md`，本文不重复；**地址格式校验**由发送页输入框通用校验逻辑覆盖，本文不重复
```

### 链特殊机制的写法

链有特殊机制（BNB Paymaster 免 Gas / Tron Energy + Bandwidth / Polkadot ED / L2 双重网络费等）时：

1. **机制描述与规则**：写到本文档对应链章节（如「BNB Chain — Paymaster 免 Gas」「Tron — 资源模型」）
2. **测试用例**：仅在主流程相关行的预期结果里**点出可观察行为**（如「网络费用 = 0 BNB（USD1 永久免 Gas）」），不在用例文档单独建机制说明章节、不写描述性 blockquote

## 📋 规则说明

### 金额转账测试规则（通用规则）

所有涉及金额转账的用例必须覆盖最大值、最小值以及中间值。

**最大值规则**：
- 定义为钱包里的余额（用户自己的可用值）
- 需要考虑账户最低余额限制
- 需要考虑 Gas 费用预留
- 需要考虑锁定金额

**最小值规则**：
- 需要根据具体的链和代币规则来定
- 不同链可能有不同的最小转账限制
- 具体规则见下方各链规则

**中间值规则**：
- 选择余额的中间值进行测试
- 确保覆盖正常转账场景

### 硬件钱包转账测试规则

#### 硬件钱包签名流程验证规则（强制）

在硬件钱包主币转账、代币转账以及 DAPP 签名时，必须验证设备上显示的签名详情：

**盲签链处理**：
- 如果该链支持盲签，则不需要校验签名详情

**非盲签链验证项**（必须验证以下信息在硬件设备上正确显示）：
- (a) From 地址与 To 地址
- (b) 转账金额
- (c) 链的名称显示
- (d) Gas 费用等相关参数
- (e) 代币的名称（代币转账时）

**验证方式**：
- 在用例的"预期结果"中明确标注需要验证硬件设备上显示的上述信息
- 确保签名详情与交易参数一致

#### 硬件钱包转账测试用例简化规则（强制）

生成硬件钱包转账测试用例时，遵循以下简化原则：

**不覆盖内容**（已在软件钱包用例中覆盖，硬件钱包用例中不再重复）：
- 新旧账户判断（如向新账户转账 vs 向老账户转账，这部分在软件钱包用例中覆盖）
- 软件相关的金额限制（如账户最低余额、Gas 预留等详细规则）
- 激活功能（如 Token 激活、账户激活等）
- 其他软件钱包已覆盖的通用规则

**仅覆盖内容**（硬件钱包测试的三个核心内容）：
- (a) 主币转账
- (b) 代币转账（含普通代币、免 Gas 代币）
- (c) DAPP 交易（连接、签名、合约交互）

**硬件用例内容边界（强制）**：硬件用例**只写需要硬件签名的操作**，App 侧操作（连接授权 / 账户选择 / UI / 历史记录 / 输入边界等）由软件用例覆盖，硬件文档不重复。详见 `docs/qa/rules/wallet-rules.md §3.2.1`。

**测试入参和测试点**（必须覆盖）：

1. **金额参数**：
   - 最大值（账户最大可用余额）
   - 最小值（链规则允许的最小转账金额）
   - 中间值（正常转账场景）

2. **Memo 参数**（如链支持 Memo）：
   - 超长 Memo（接近或达到链的最大长度限制）
   - 短 Memo（如 1-5 个字符）
   - 不同格式的字符串（纯数字、包含字符、特殊字符等）
   - **注意**：不同链的 Memo 最大长度不同，需参考下方各链规则（如 Stellar 链 Memo 最大长度为 28 字节）

3. **硬件显示验证**：
   - 验证硬件设备上签名详情的显示是否正常
   - 验证信息清晰可读
   - 验证显示内容与交易参数一致

**预期结果要求**：
- 必须包含硬件设备上显示的签名详情（From/To 地址、转账金额、链名称、Gas 费用、代币名称、Memo 等）
- 必须包含交易提交后需要确认的信息（交易成功、余额扣除、交易记录生成等）
- 硬件确认流程验证直接整合在转账用例的预期结果中，不需要单独编写硬件确认流程章节

**用例设计原则**：
- 聚焦硬件钱包特有的验证点（签名流程、设备显示）
- 避免与软件钱包用例重复
- 保持用例简洁，突出硬件钱包测试重点
- 使用组合入参形式（金额 × Memo）进行测试，提高测试覆盖率

#### 硬件转账用例生成规则（强制）

生成硬件钱包转账测试用例时，**主币转账和代币转账各生成 3 条用例**，共 6 条用例。

**用例组合规则**（memo + 金额）：

**主币转账三条用例**：
1. **最小值 + memo（数字字符组合，最大限制）**
   - 金额：最小转账金额（参考下方各链规则）
   - Memo：数字字符组合，长度达到链的最大长度限制（参考下方各链规则，如 Stellar 链为 28 字节）
2. **最大值 + 不加 memo**
   - 金额：最大值（余额 - 账户最低余额 - Gas，参考下方各链规则）
   - Memo：不填写
3. **中间值 + 纯数字 memo**
   - 金额：中间值（余额的中间值）
   - Memo：纯数字（自动推断为对应类型，参考下方各链规则，如 Stellar 链为 MEMO_ID 类型）

**代币转账三条用例**：
1. **最小值 + memo（数字字符组合，最大限制）**
   - 金额：最小转账金额
   - Memo：数字字符组合，长度达到链的最大长度限制（参考下方各链规则，如 Stellar 链为 28 字节）
2. **最大值 + 不加 memo**
   - 金额：最大值（代币余额）
   - Memo：不填写
3. **中间值 + 纯数字 memo**
   - 金额：中间值（代币余额的中间值）
   - Memo：纯数字（自动推断为对应类型，参考下方各链规则，如 Stellar 链为 MEMO_ID 类型）

**注意事项**：
- 不同链的 Memo 最大长度不同，需参考下方各链规则（如 Stellar 链 Memo 最大长度为 28 字节）
- 不同链的 Memo 类型推断规则不同，需参考下方各链规则
- 如果链不支持 DApp 交易，不输出 DApp 相关用例（需参考下方各链规则中的硬件钱包支持情况）

## 🔗 各链转账规则

### Stellar (XLM)

#### 账户最低余额限制
- **基础保留**：账户余额不能低于 **1 XLM**
- **Token 占用**：每激活（Add Trustline）一个 Stellar Asset Token，必须锁定 **0.5 XLM**
- **示例**：账户持有 10 个 Asset Token，则有 5 XLM 被锁定不可转出

#### 主币转账（XLM）
- **向新账户转账**：金额必须 ≥ **1 XLM**（否则交易失败）
- **向老账户转账**：无最小金额限制（但需预留 Gas）
- **Gas 费用**：普通交易费用极低（默认约 100 stroops）

#### 代币转账
- **Stellar Asset Token**：
  - 接收方必须已激活（建立 Trustline）该 Token
  - 如果接收方未激活，交易应失败或前端提示无法转账
  - 精度：固定 7 位
- **Contract Token**：
  - 无需接收方激活
  - 精度需通过链上查询
  - 包含 Resource Fee，费用较高

#### Memo 规则
- **自动推断**：
  - 输入纯数字 → 自动设为 MEMO_ID 类型
  - 输入包含字符/字符串 → 自动设为 MEMO_TEXT 类型
- **不支持**：Hash 和 Return 类型的 Memo
- **最大长度**：Memo 最大长度为 **28 字节**
- **CEX 充值**：往 Binance 和 OKX 充值 XLM，务必填写 Memo

#### 硬件钱包支持
- ✅ XLM 转账：支持
- ✅ Asset Token 转账：支持
- ❌ Contract Token 转账：不支持
- ❌ Swap：不支持
- ❌ Soroban Token：不支持
- ❌ DApp 交易：不支持（Stellar 硬件钱包不支持 DApp 交易，不输出 DApp 相关用例）

### Lightning Network (BTC)

> 闪电网络是链下协议，与其他链在金额单位、历史记录字段、费用模型上存在显著差异，生成用例前必须先阅读本节。

#### 单位与精度
- **最小单位**：1 sat（satoshi），**整数**；1 BTC = 100,000,000 sats
- **单笔上限**：100,000,000 sats（1 BTC）；超出提示「金额不得超过 100000000 sats」
- **法币换算**：法币金额 ↔ sats 可双向切换；法币过小时可能显示为 0
- **法币输入 → sats 必须四舍五入为整数**（强制）：
  - 输入法币金额时，换算出的 sats **必须显示为整数**，不得出现小数
  - 若显示非整数 sats（如 `123.6 sats`），交易将失败（链下协议不支持非整数 sats）
  - 确认页、历史记录中的金额同样必须为整数 sats

#### 金额与余额规则
- **指定金额发票**（BOLT11 携带金额）：
  - 金额自动填入，**输入框不可修改**
  - 若发票金额 + 1% 浮动费用 > 账户余额，禁止提交，提示「余额不足！请至少预留 invoice 金额的 1% 来应对潜在的交易费浮动」
- **未指定金额发票 / LN Address / LNURL**：
  - 金额 = 0 或空：禁止提交，提示输入金额
  - 金额 > 1 BTC 或超过可用余额：禁止提交
  - 金额最小值 1 sat；点击「Max」按钮 = 可用余额

#### 收款地址类型（3 种）
| 类型 | 格式 | 示例 |
| --- | --- | --- |
| 发票（BOLT11） | `lnbc...` / `lntb...` | `lnbc300u1pj2...` |
| LN Address | `user@domain` | `shortmen02@walletofsatoshi.com` |
| LNURL | `lnurl1...` | `lnurl1dp68gurn8...` |

#### 发票描述（Memo / 备注）
- **最大长度**：40 字符（含空格、换行、特殊字符、Emoji）
- **超长**：禁止提交，提示「描述不得超过 40 字符」
- **未填写默认值**：`OneKey Invoice`
- **对端携带**：由开票方写入，收款方不可改

#### 网络费规则
- 交易费 = 0 或由路由决定（发票携带）
- **不可修改、不可切档**（无 Fast / Standard / Slow）
- 每次交易金额可能不同（不固定）

#### 历史记录特性（与其他链的关键差异）
| 字段 | 闪电网络 | 其他链 |
| --- | --- | --- |
| 交易哈希跳转 | **无跳转**（闪电网络无区块浏览器） | 可跳转区块浏览器 |
| 交易对象 From/To | **仅展示当前账户** | 同时展示双方 |
| 网络费调节 | **不可修改** | 通常可切档 |
| 网络标识 | `Lightning` / `闪电网络` | 各链独立名称 |
| 交易完成时长 | 秒级 | 分钟级 |
| 主币余额 = 0 发送前置拦截 | **不生效** | 单链视图下生效 |

#### 异常发票处理
| 发票状态 | 预期提示 |
| --- | --- |
| 已过期 | 地址无效 |
| 已被支付过 | 发票已支付 |
| 格式错误（截断 / 缺前缀） | 地址无效 |

#### 账户与选择地址
- **「我的账户」Tab 不支持闪电网络**：发送流程中选择地址弹窗仅显示「最近」Tab（参考 `wallet-rules.md §5.05`）
- **账户间数据隔离**：多个闪电账户的余额、历史记录独立，切换账户后刷新为对应账户数据

#### HW 钱包支持
- ✅ 主币转账（发票 / LN Address / LNURL）：支持
- ✅ dApp 支付（LApp Deposit 等发送类交易）：支持
- ❌ 代币转账：闪电网络无代币概念

#### HW 钱包签名范围（生成 HW 用例时必须遵守）
| 操作 | 是否需要 HW 签名 | 说明 |
| --- | --- | --- |
| 发送（发票 / LN Address / LNURL） | ✅ 需要 | HW 设备按「发送 / Deposit 签名页字段」显示（见下表） |
| 接收（生成发票） | ❌ 不需要 | App 侧生成 |
| dApp 连接授权（App 弹出授权弹窗） | ❌ 不需要 | App 内完成，不涉及 HW |
| dApp 支付（LApp Deposit） | ✅ 需要 | 属发送类交易，HW 签名页字段同发送 |
| dApp 赎回（LApp Withdraw） | ❌ 不需要 | 属接收类 |
| dApp 授权登录（LNURL-auth：Alby / WebLN / Nostr Wallet Connect） | ✅ 需要（HW 钱包场景） | HW 钱包需 HW 签名 linkingKey；HD 钱包在 App 内完成不涉及 HW。HW 签名页字段见下表「LNURL 授权签名页字段」 |
| 账户选择 / App UI 操作 | ❌ 不需要 | 与 HD 钱包一致 |
| 通道开启 / 关闭 | ✅ 需要 | 涉及链上交易 |

#### HW 设备型号签名页字段差异（所有 HW 用例默认回归所有型号）

| 型号 | 发送 / Deposit 签名页字段 | LNURL 授权签名页字段 |
| --- | --- | --- |
| Mini | 发送方地址；消息内容（创建时间 / 过期时间 / 发票等） | 标题 = `LNURL 授权`；域名；展示数据 |
| Classic / Classic 1S | 标题 = `签署 Bitcoin 消息`；发送方地址；消息内容（创建时间 / 过期时间 / 发票等） | 标题 = `批准 LNURL 授权`；域名；展示数据 |
| Touch / Pro | 标题 = `签署 Bitcoin 消息`；消息内容（创建时间 / 过期时间 / 发票等）；发送方地址 | 展示消息数据；域名 |

> 断言口径：每行列出的字段均需在 HW 屏上可见，内容与 App 侧推送一致。HW 用例无需为每个型号拆独立用例，在「预期结果」中以「按设备型号字段差异表逐型号校验」统一描述即可。

#### HW 钱包用例范围约束
- HW 钱包用例**仅覆盖需要 HW 签名的操作**，其他 App 侧操作（连接授权、UI、历史记录、账户选择等）由 HD 钱包用例覆盖，HW 文档**不重复描述**
- HW 用例预期聚焦签名屏（发送方 / 金额 / 链名 / **发票描述** / 网络费）+ 上链 + 余额
- 不需要在 HW 用例中验证 App 侧的确认页字段、历史记录详情、UI 无异常等

#### dApp（LApp）交互
- **连接协议**：Nostr Wallet Connect / Alby / WebLN
- **典型 dApp**：LNBlackjack（Deposit / Withdraw）、LNMarkets（LNURL-auth 授权登录）
- **支付时确认页必须显示**：dApp Origin、发票描述、金额、手续费、发送方 = 自己
- **LNURL 授权登录（LNURL-auth）**：
  - HD 钱包：在 App 授权页内完成（无设备交互）

  - 站点域名断言以第三方服务实际下发为准（如 LNMarkets = `api.lnmarkets.com`）

### BenFen (BFC)

> 适用于 BenFen 主网，主币为 **BFC**，OneKey 已接入。

#### 主币与精度

- **主币**：BFC，精度 **9 位小数**（最小可见单位 `0.000000001 BFC`）。
- **代币**：常见包括 **BUSD**（链上稳定币）、**BF_USDC**（USDC on BenFen）、**LONG** 等。

#### 金额规则

- **0 金额禁止**：金额 = 0 时禁止提交，「预览」按钮置灰。
- **最大值**：`Max = 余额 − 当前预估网络费用`；如有账户最低保留要求，需相应扣除（具体以链运行时为准）。
- **极小精度**：输入 `0.000000001` 等接近最小单位的金额需正常显示，无科学计数异常。

#### 网络费用

- Gas 以主币 BFC 结算；代币（BUSD / BF_USDC / LONG 等）转账的网络费同样以 BFC 结算，与代币余额分离展示。
- 是否支持多档切换、是否提供进阶设置以产品实际为准（首次接入需对照截图核对，本规则不强制声称）。

#### Memo / Tag

- BenFen 主币与代币转账**协议层不携带 Memo**；OneKey 发送页不提供 Memo 输入框。

#### 历史记录字段

- 标准字段：金额、类型、状态、时间、from / to、交易哈希、网络费用、网络。
- 是否额外显示区块高度 / 确认数 / Nonce 等以链与产品实现为准（首次接入需对照截图核实）。

### EVM 通用（含 BNB Chain）

> 适用于 Ethereum、BNB Chain、Polygon、Avalanche、Arbitrum、Optimism、Base 等基于 EVM 的网络。本节聚焦各 EVM 链共享的转账行为；个别链特性（如 BNB Chain Paymaster 免 Gas）单独标注。

#### 地址与跨链复用

- **地址格式**：0x 开头 42 位（含 0x），各 EVM 链共享同一地址（同一私钥派生出同一地址）。
- **地址识别四类提示**（发送页输入地址后展示）：
  - **钱包内地址**：接收方为本钱包派生出的另一账户地址 → 显示账户名标签
  - **地址簿地址**：接收方在地址簿中存在条目 → 显示地址簿标签
  - **首次转账**：接收方在所有 EVM 链上**从未**收到过本账户的转账
  - **链首次转账**：接收方在其他 EVM 链上有过交互，但**当前链**上从未收到过本账户的转账（多链共享地址特有）
- **地址格式校验**：发送页输入框对长度、十六进制字符、校验和（EIP-55 大小写）有通用校验，本节规则文档不重复。

#### 网络费用与进阶设置

- **四档可切换**：Slow / Standard / Fast / Custom（具体文案以产品为准）；切换后**预估费用**与**预估时间**同步联动。
- **进阶设置**支持修改：Gas Limit、Gas Price（或 EIP-1559 的 Max Fee / Max Priority Fee）、Nonce、Hex Data。
- 与 Tron 等固定档位链对比：EVM 必须支持档位调节，**不得隐藏档位入口**。

#### Nonce 与历史记录字段

- **Nonce**：交易确认页与历史记录详情**必须显示** Nonce 值，并允许进阶设置修改。
- **历史记录详情字段**：金额、类型、状态（处理中 / 成功 / 失败）、时间、from / to、交易哈希、网络费用、网络、区块高度、**Nonce 值**、确认数；点击交易哈希可跳转区块浏览器对账。

#### Memo / Tag

- EVM 主币转账**不携带 Memo / Tag**；OneKey 发送页不提供 Memo 输入框。
- ERC20 / BEP20 代币（USDT / USDC / USD1 等）同样无 Memo 概念。

#### EVM L2（Optimism / Base / Arbitrum 等）

- **主币**：均为 **ETH**（Gas 也以 ETH 结算）。
- **网络费用模型**：L2 网络费 = **L2 执行费 + L1 数据费（calldata 上链费用）**。
  - L1 数据费随 L1（Ethereum 主网）实时 Gas 波动，可能短时变化较大；
  - 确认页应显示**预估总费用**；如产品提供 breakdown，需正确拆分 L1 / L2。
- **EIP-1559 支持**：Optimism / Base / Arbitrum 均支持 1559 字段（Max Fee / Max Priority Fee）。
- **Max 计算**：`Max = ETH 余额 − 当前档位预估总费用（含 L1 + L2）`；切换费用档位后 Max 同步刷新。

---

#### BNB Chain — Paymaster 免 Gas（链特有）

- **USDT 免 Gas**：每个账户**每天前两笔** USDT 转账免 Gas（手续费显示 **0 BNB**），生效条件为**转账金额 > 0.1 USDT**。
  - 第 3 笔及之后：恢复正常 BNB 手续费
  - 金额 ≤ 0.1 USDT：不享受免 Gas，正常 BNB 手续费
- **USD1 免 Gas**：**不限金额、不限笔数、不限日期**，所有 USD1 转账永久免 Gas（手续费始终显示 0 BNB）。
- **主币（BNB）转账**：不享受免 Gas，按常规四档收费。

### Tron (TRX)

> 适用于 Tron 主网，主币为 TRX，代币标准 TRC10 / TRC20 / TRC721；OneKey 已接入。

#### 账户与地址

- **地址格式**：Base58Check，以 `T` 开头，长度 34；十六进制内部表示以 `41` 开头。
- **新账户激活**：向**未激活**的链下地址转 TRX 时，发送方需支付约 **1 TRX 激活费**（或等额带宽 / 能量），具体以链上策略为准；激活后账户在链上正式创建。
- **转账给自己拦截**：发送方与接收方为同一地址时，前端必须拦截，提示「不能转账给自己」类错误。

#### 资源模型（Energy + Bandwidth）

- **Bandwidth（带宽）**：每笔交易体积消耗；账户每天免费 **≈ 1500 带宽**。
- **Energy（能量）**：执行智能合约时消耗（如 USDT-TRC20 转账触发合约调用）。
- **资源来源**：
  - 免费配额（每日重置）
  - 抵押 / 质押 TRX 获得能量与带宽
  - 资源不足时**烧 TRX 抵扣**
- **TRX 普通转账**：通常仅消耗带宽，不消耗能量。
- **TRC20 代币转账（如 USDT）**：必须同时消耗能量 + 带宽；能量不足时烧 TRX。

#### 网络费用 / 档位

- **固定档位**：Tron App 内不提供多档可调网络费（无 Slow / Fast / Custom），统一显示**预估手续费**。
- **资源抵扣展示**：交易确认页须显示「能量带宽抵扣数据」（实际消耗的能量与带宽数量），供用户核对。
- **TRX 烧费**：当资源不足、需要烧 TRX 时，确认页费用展示应包含烧 TRX 等价值。

#### Memo / 备注

- Tron 主币转账与 TRC20 代币转账**协议层不携带 Memo**；OneKey 转账主流程**不提供 Memo 输入框**。
- 充值交易所（Binance / OKX 等）TRX、USDT-TRC20 通常**无需 Memo**（按 CEX 文档执行）。

#### 历史记录字段（Tron 特有）

| 字段 | 说明 |
| --- | --- |
| 资源 | 扣除的能量 + 带宽（含烧 TRX 等价） |
| 网络费用 | 固定档位（预估值） |
| 区块高度 | 上链区块号 |
| 确认数 | 当前确认次数（实时刷新） |
| 交易哈希 | 可点击跳转区块浏览器 |
| 网络 | 显示「Tron」 |

#### 硬件钱包支持

- ✅ TRX 转账：支持
- ✅ TRC20 代币转账（USDT 等）：支持
- ✅ DApp 交易：支持（具体以产品列表为准）

### Polkadot / Kusama 系（Substrate）

> 适用于 Polkadot AssetHub、Kusama AssetHub、Astar、Manta Atlantic、Joystream、Hydration、Bifrost Polkadot、Bifrost Kusama 等基于 Substrate 的网络（以 OneKey 已接入列表为准）。

#### 账户与地址（SS58）

- **地址格式**：SS58 编码，不同链有不同前缀字节，导致同一公钥在各链显示为不同地址。
- **跨链复用**：同一助记词派生的账户在 Polkadot 系各链上为不同的地址展示，但底层私钥一致。
- **格式校验**：地址必须通过 SS58 校验和，错误地址 / 非本链前缀必须前端拦截。

#### 存活金额（Existential Deposit, ED）

- **概念**：账户余额低于 ED 后会被链上「Reap」（账户清零、地址记录销毁）。
- **不同链 ED 不同**：Polkadot AssetHub ≈ 0.1 DOT、Kusama AssetHub ≈ 0.000033 KSM、Astar ≈ 0.000001 ASTR、Manta Atlantic ≈ 0.1 MANTA、Hydration ≈ 0.001 HDX、Joystream ≈ 0.0000266 JOY、Bifrost ≈ 0.01 BNC（具体以链上运行时配置与产品口径为准，定期校对）。
- **向新地址转账金额 < ED**：发送页**金额输入框下方**红字提示「**收款方需要预留 X {主币} 才能激活**」（X 为该链 ED 数值），「预览」按钮置灰，禁止提交。
- **Max 转账**：钱包侧 Max 必须为 `余额 - Gas - ED`（确保发送后账户保留 ED；如果产品允许「全额清空发送」需在 UI 二次确认）。
- **新地址边界测试归属**：新地址 < ED / 余额清零等边界用例集中在 `docs/qa/testcases/cases/wallet/2026-04-09_Wallet-发送-转账规则&租金.md` §3，链专项用例不重复。

#### 金额与精度

- **最大值**：钱包余额上限（须预留 Gas + ED，参考上一节）。
- **最小值**：链支持的最小精度单位（由各链 decimals 决定，但实际可发送的最小值不应低于 ED 对应数量级）。
- **禁止 0**：金额 = 0 时禁止提交。
- **代币转账**（适用 AssetHub / Hydration 等支持 Assets pallet 的链）：
  - AssetHub 系（Polkadot / Kusama AssetHub）支持 USDT / USDC 等 Assets pallet 资产；
  - Hydration、Astar 等支持其原生代币 / LP 代币；
  - 代币的 ED / 最低保留逻辑可能与主币不同，按链配置与产品口径执行。

#### Memo / Remark

- **协议层**：Substrate 通过独立 `system.remark` extrinsic 携带备注，不属于 `balances.transfer` 字段。
- **OneKey 实现**：当前不在转账主流程提供「Memo / Remark」输入框（以产品实际为准）；如某链特殊支持，按该链单独说明。
- **CEX 充值**：交易所通常不依赖 Substrate Memo 区分用户，按 CEX 文档要求执行（一般无需 Memo）。

#### Gas / 费用

- 费用为 Substrate Weight × 单价，主币结算；预览页应显示预估费用并支持多档（如产品提供）。
- 不同链 Gas 量级差异大（如 Astar、Joystream 极低，Polkadot AssetHub 相对较高），UI 法币换算需正确处理小数位。

#### 硬件钱包支持

- ✅ 主币转账：支持
- ✅ AssetHub Assets pallet 代币转账：支持（具体以产品列表为准）
- ❌ DApp 交易：暂不支持（具体以产品列表为准；不输出 DApp 用例时按规则简化）

### Cosmos 系（Cosmos SDK / IBC）

> 适用于 Akash、Cosmos Hub、Cronos POS Chain（原 Crypto.org）、Fetch.ai、Juno、Osmosis、Secret Network、Celestia、Babylon、Noble 等基于 Cosmos SDK 的网络（以 OneKey 已接入列表为准）。

#### Memo / 备注

- **最大长度**：**512 字节**（按 UTF-8 编码占用字节数计算）。
- **字符类型**：允许空备注；允许纯数字、字母、中文、Emoji、混合文本等（不限制字符形式）。
- **超限处理**：超过 512 字节时**禁止提交**。
- **边界测试**：必须包含「恰好 512 字节」与「513 字节（或超过上限）」两类用例。

#### 金额与精度

- **最大值**：当前账户在该网络上的**余额上限**（最大可转金额；需预留本链 Gas，具体预留以产品/链实现为准）。
- **最小值**：该币种支持的**最小精度单位**（由代币 decimals 决定）。
- **禁止 0**：金额 = 0 时**禁止提交**。
- **代币类型**：需同时覆盖**网络主币**与 **IBC / 原生稳定币等代币**（如 Osmosis 上 ATOM、Noble 上 USDC）。
- **IBC 代币**：支持 IBC 代币转账（链上资产标识常见为 `ibc/<hash>`），用例需覆盖至少 1 个 IBC 代币。

#### Gas / 费用

- 预览页需支持**多档网络费用**切换；切换后预估费用与广播前展示一致（与链上实际扣费允许合理波动，须在区块浏览器可核对）。

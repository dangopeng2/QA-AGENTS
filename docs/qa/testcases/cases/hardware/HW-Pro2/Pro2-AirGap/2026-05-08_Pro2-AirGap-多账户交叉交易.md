# Pro2 - AirGap 多账户交叉交易（转账 + 消息签名 + DApp Swap）

> 规则文档：`docs/qa/rules/hardware-rules.md`、`docs/qa/rules/swap-rules.md`
> 测试端：MetaMask + OneKey App + Pro2 设备端
> 关联：Jira OK-4818
> 变更说明：按 **Passphrase 阶段 + 地址分组** 重组（标准钱包阶段：A1/A2；密语钱包阶段：B1/B2）；补充错误密语三类边界；OctaSpace 已确认为 Legacy 链；A/B 4 个地址在 MetaMask 中并存无需断开
>
> **关键说明**：Pro2 标准钱包的「账户 1」和密语钱包的「账户 1」是**不同派生路径**得到的不同地址，导入 MetaMask 后是**两组独立地址**。本文用 **A1/A2**（标准钱包前两个地址）+ **B1/B2**（密语钱包前两个地址）区分。

---

## 前置条件与测试数据

1. Pro2 设备已激活，并已开启「AirGap」开关
2. 通过 AirGap 协议二维码导入到 MetaMask 的 4 个地址：
   - **A1 / A2** —— Pro2 关闭 Passphrase 时导出的前两个 EVM 地址
   - **B1 / B2** —— Pro2 切到 Passphrase = `onekey` 时导出的前两个 EVM 地址（与 A1/A2 不同）
3. 4 个地址在以下链上**均有可用余额**（满足 Gas + 测试金额）：
   - **Arbitrum One** — L2 交易
   - **OctaSpace** — Legacy 普通交易（OneKey App 已支持，Legacy 类型 ✓）
   - **Polygon** — EIP-1559 交易
4. Passphrase 测试用密语固定为：**`onekey`**
5. DApp 准备：
   - 消息签名（Personal Sign / Typed Data v3 / v4）：`https://metamask.github.io/test-dapp/`
   - DApp Swap：`https://www.sushi.com/swap`
6. 字段核对口径：
   - **交易类**（L2 / Legacy / 1559 / Swap）：核对**发送方 / 接收方 / 金额 / Data / 手续费**
   - **消息签名类**（Personal Sign / Typed Data v3 / v4）：核对**签署消息的完整文本内容**
7. 转账测试金额（dust 级，能上链即可）：主币 `0.0001`、ERC-20 `0.01`；DApp Swap 用最小可成交单位
8. **MetaMask 账户共存**：A1/A2 与 B1/B2 在 MetaMask 中**同时存在为 4 个独立账户，无需断开**；切换阶段时只需在 MetaMask 内切到对应地址即可
9. **执行顺序**：第 1 节（标准钱包阶段，A1/A2）→ 第 2 节（密语钱包阶段，B1/B2），跨阶段时在 Pro2 端切换 Passphrase 状态

## 3. 错误密语边界

> **统一断言**：本节所有错误密语场景，Pro2 端均**提示「无效的交易」**，不生成签名二维码。

| 优先级 | 场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| ❗️❗️P0❗️❗️ | 密语错误（输入与 MetaMask 当前地址不匹配的密语） | 1. MetaMask 切到 B1，发起任一签名<br>2. Pro2 扫码后，**在签名前的密语输入页输入错误密语**（如 `wrongpass`）<br>3. 提交密语进入签名页并点击「确认」 | **提示「无效的交易」**，不生成签名二维码 |
| P1 | 密语为空 | 1. MetaMask 切到 B1，发起任一签名<br>2. Pro2 扫码后密语输入框留空，直接提交（如允许） | **提示「无效的交易」** 或前端拦截「密语不能为空」 |
| P1 | 用户在密语输入页主动取消 | 1. MetaMask 切到 B1，发起任一签名<br>2. Pro2 扫码后在密语输入页点击「取消」 | Pro2 退出密语输入并返回主屏幕，不生成签名；App 端二维码弹窗保持等待 |

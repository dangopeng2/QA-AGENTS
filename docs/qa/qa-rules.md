## 4）A. 通用检测维度（默认必须覆盖）
1. 输入与表单（Input & Forms）
   - 边界：空 / 空格 / 超长 / 特殊字符 / Emoji
   - 格式：精度、负数、千分位粘贴、自动 Trim
   - 安全：XSS / 注入尝试、敏感词过滤
2. 列表与展示（List & Display）
   - 空状态、分页 / 加载、排序 / 筛选
   - 实时刷新（WebSocket / 轮询）
   - 多语言 / 暗黑模式 / 长文本截断
3. 状态流转与逻辑（State & Logic）
   - 状态闭环（Pending / Success / Fail）
   - 断网 / 杀进程恢复
   - 并发点击、防抖、权限 / 未登录
4. 数值与资产计算（Calculation & Assets）
   - 精度 / 科学计数法 / 最小单位
   - 前端展示 vs 实际扣除一致性
   - 临界点（刚好够 / 刚好不足）
5. 网络与外部依赖（Network & External）
   - 弱网 / 超时 / 重试
   - 404 / 500 / 空数组容错
   - 权限拒绝（相机 / 剪贴板）引导
6. 性能（Performance）
   - 高频操作响应
   - 长列表 FPS
   - 页面反复进入内存释放
   - **Web 端专项**（见 `docs/skills/specs/performance.md`）
7. 兼容性（Compatibility）
   - 小屏 / 刘海屏 / 折叠屏
   - 系统字体缩放
8. 安全（Security）
   - 敏感信息掩码
   - 关键操作二次确认 / 生物识别
9. 边界值测试（Boundary Testing）
   - 详见下方 4.1 节「边界值测试规范」

### 4.1 边界值测试规范（强制覆盖）

> 所有涉及用户输入的功能，必须覆盖边界值测试场景。

#### 输入类型边界值

| 类型 | 边界值 | 测试点 |
|------|--------|--------|
| 数值 | 0、-1、最大值、最小值、小数精度 | 负数处理、零值处理、超大值溢出、精度截断 |
| 字符串 | 空、空格、超长、特殊字符 | 空字符串、纯空格、Emoji、SQL 注入、XSS 尝试 |
| 地址 | 无效地址、空地址、格式错误 | 校验错误提示、格式容错 |
| 枚举 | 边界值、非法值 | 超出枚举范围的处理 |

#### 金额边界值（资产类必测）

| 场景 | 测试值 | 预期结果 |
|------|--------|---------|
| 负数金额 | `-1` | 校验失败，显示错误提示 |
| 零金额 | `0` | 校验失败或显示告警 |
| 超大金额 | `999999999999` | 余额不足告警 |
| 极小精度 | `0.000000001` | 正确处理或精度告警（按链规则） |
| 刚好够 | `余额 - Gas` | 交易成功 |
| 刚好不足 | `余额 - Gas + 0.01` | 余额不足提示 |

#### 字符串边界值

| 场景 | 测试值 | 预期结果 |
|------|--------|---------|
| 空字符串 | `""` | 必填校验失败 |
| 纯空格 | `"   "` | Trim 后校验失败或提示 |
| 超长字符串 | 超过最大长度限制 | 截断或显示长度错误 |
| 特殊字符 | `<script>alert(1)</script>` | XSS 过滤，正常显示或拒绝 |
| SQL 注入 | `' OR '1'='1` | 正常处理，无注入风险 |
| Emoji | `🚀💰` | 正确显示或过滤 |

#### Memo/Tag 边界值（转账类必测）

| 场景 | 测试值 | 预期结果 |
|------|--------|---------|
| 空 Memo | 不填写 | 正常提交（如链允许） |
| 最大长度 | 达到链的最大限制（如 Stellar 28 字节） | 正常提交 |
| 超出最大长度 | 超过链的最大限制 | 显示长度错误或截断 |
| 纯数字 | `123456789` | 按链规则自动推断类型 |
| 数字+字符 | `abc123` | 按链规则自动推断类型 |

- 功能覆盖：主币转账、代币转账、NFT、法币出入金、UTXO 管理
- **详细规则**：`docs/qa/rules/wallet-rules.md`

### Swap
- 状态机：询价→路径→授权→Swap→Pending→Success/Fail
- 资金流：授权额度、余额对账、手续费/滑点展示一致
- 风控：高滑点提醒、合约风险提示、最小收到保护
- **详细规则**：`docs/qa/rules/swap-rules.md`
- **网络特性**：`docs/qa/rules/swap-network-features.md`
- **渠道/场景 Markdown 用例**（同链跨链分文件、P0-only、金额边界并入 §1、专项文档默认不收异常/账户限制章）：见 `swap-rules.md` 中「Swap 渠道/场景用例文档编写规范」

### Market
- 数据一致性：K 线/列表/详情价格一致
- 状态机：WS 断连→重连→补数据→防闪烁
- 性能：长列表虚拟化、K 线交互帧率
- Token 详情页 K 线（周期、OHLC、指标、十字光标/移动端点击、第三方对照）：`docs/qa/requirements/Market-图表.md`
- **详细规则**：`docs/qa/rules/market-rules.md`

### DeFi
- 状态机：授权→存入→计息→领取→赎回
- 资金流：本金/收益拆分、APR/APY 口径、T+N 到账
- 风控：高风险池提示、额度限制、重复操作保护
- **接口 / Apifox 测试数据**：Pendle 相关 Earn 接口的请求参数 `vault`（合约地址）须为**全小写** hex；Apifox 易改写裸地址，可用集合变量 `{{pendle_vault_*}}` 引用小写值。**仅 Pendle 适用**。见 `docs/qa/rules/defi-rules.md` §2.15。
- **详细规则**：`docs/qa/rules/defi-rules.md`

### Perps（合约）
- 状态机：开仓→加仓→减仓→平仓→反手；挂单/撤单；TP/SL
- 资金流：保证金/可用余额、资金费率结算、强平后残值
- 风控：维持保证金率预警、爆仓保护、价格保护
- **详细规则**：`docs/qa/rules/perps-rules.md`

### Hardware（硬件钱包）
- 状态机：连接→配对→验证→操作→确认→断开
- 设备管理：设备列表、设备详情、设备设置、固件更新
- 兼容性：多设备型号、多连接方式（USB/蓝牙/QR）
- **输出规则**：禁止输出自动化相关内容
- **详细规则**：`docs/qa/rules/hardware-rules.md`

### Referral（返佣）
- 状态机：绑定→生效→累计→结算→发放
- 资金流：累计收益口径、T+N、精度与四舍五入
- 数据一致性：跨端一致，邀请人与被邀请人奖励对应
- **详细规则**：`docs/qa/rules/referral-rules.md`

### Browser（DApp）
- 状态机：connect→请求权限→签名/交易→确认/拒绝→回传
- 安全：origin 展示、钓鱼检测、盲签阻断
- 兼容：多 tab、EIP-6963、多钱包冲突
- **详细规则**：`docs/qa/rules/browser-rules.md`

### Utility（通用）
- 设置：语言、主题、通知偏好、安全设置
- 通知：推送→到达→点击跳转→已读
- 风控：设备检测、交易拦截、隐私保护
- 通用功能：地址簿、扫码、搜索、App 升级
- **详细规则**：`docs/qa/rules/utility-rules.md`

### Prime

- **详细规则**：`docs/qa/rules/prime-rules.md`

### 7.3 每个场景输出表格（固定格式）
使用 **Markdown 表格**，单元格内多行使用 `<br>` 标签换行。

表头固定为：
`| 优先级 | 场景 | 操作步骤 | 预期结果 |`

**列说明**：
- **优先级**：❗️❗️P0❗️❗️ / P1 / P2（P0 用例必须在前后加 ❗️❗️ 标识）
- **场景**：当前状态或前置条件（简洁描述当前测试的起始状态）
- **操作步骤**：一个明确动作
- **预期结果**：可观察的结果（仅使用允许词表中的词汇）

规则：
- 同场景可多行（P0 / P1 / P2）。
- **关键格式**：**场景、操作步骤、预期结果** 三列，只要单元格内包含多行内容，都必须使用 `<br>` 标签分隔，并且必须使用序号（1. 2. 3. 4.）标识每一行。
- **序号格式示例**：
  - 场景列：`1. 设备已安装 OKX App<br>2. 使用硬件钱包`
  - 操作步骤列：`1. 进入借贷首页<br>2. 观察页面展示`
  - 预期结果列：`1. 显示 6 个代币<br>2. 每个代币右侧显示复选框<br>3. 底部显示按钮`
- 不同测试场景禁止合并为同一张表。
- **P0 用例排序规则（强制）**：如果同一表格内的测试步骤之间没有前后依赖关系，P0 用例必须排在表格顶部，优先级顺序为 P0 → P1 → P2。仅当用例之间存在明确的执行依赖（如"先添加指标，再删除指标"）时，才可按操作流程顺序排列。

**格式示例**：

| 优先级 | 场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| ❗️❗️P0❗️❗️ | 1. 已连接钱包<br>2. 钱包余额为空 | 1. 进入借贷首页<br>2. 观察页面展示 | 1. 显示空状态文案："钱包为空，请先接收、跨链或购买资产"<br>2. 显示「接收」和「跨链」按钮 |

### 7.3.1 钱包账户类型分组规则（强制）

**核心原则**：新链接入按钱包账户类型（软件钱包/硬件钱包）分成两条用例输出，同一类型的所有功能测试集中在一起。

**分组方式**：

1. **软件钱包分组**（必须集中）：
   - 软件钱包主币转账
   - 软件钱包代币转账（含普通代币、免 Gas 代币）
   - 软件钱包 dApp 连接与交易
   - 软件钱包私钥/助记词导出
   - 软件钱包账户管理

2. **硬件钱包分组**（必须集中，仅输出以下三项）：

## 9）生成后自动执行：合并 & 去重（强制）
### 合并规则
同一测试场景下：
- 输入数据一致，或
- 操作步骤一致，仅校验点不同
则必须合并为一条（或一行）用例描述，避免重复。

### 去重规则
- 删除语义重复的预期
- 删除隐含重复步骤
- 同一校验点只保留一次

### 禁止合并
- 不同测试场景
- 不同业务状态
- 不同优先级（P0 不得与 P1/P2 合并）

### 结果自检
- 合并后仍可人工执行
- 覆盖资金 / 风控 / 状态切换
- 确认表格中使用了 `<br>` 且无超链接样式
- 预期结果仅使用允许词表中的词汇

**② 确认现状**
- 确认当前 UI 已有哪些引导/提示/保护机制（如 placeholder、风险提示、确认页核对）
- 不确定的标注"建议前提：如果当前没有 X"
- 不确认就不提

**③ 分析建议**
- 从以下维度思考（按优先级排列，无内容的维度省略）：
  1. **安全风险**（资金安全、隐私泄露、误操作导致不可逆后果）
  2. **操作效率**（高频操作路径是否偏长、重复步骤是否可简化）
  3. **易用性**（术语是否难懂、交互是否符合直觉）
  4. **信息层级**（关键信息是否突出、次要信息是否干扰）

**④ 自检过滤（每条建议必须过以下检查）**
- 前提是否正确？（数据来源、业务边界是否搞清楚了？）
- 该功能的 UI 是否已解决了这个问题？
- 这条建议实现后会不会引入新的安全风险？
- 是否在建议暴露有意隐藏的安全限制？（如建议把观察账户显示为灰色）
- 是否在过度解释用户主动开启的功能？（有开关 = 用户已理解）
- **任一检查不通过，删除该条建议**

**⑤ 输出**
- 格式见下方「输出格式」
- 位置：用例文档末尾，用 `---` 分隔，独立于测试表格

**⑥ 用户反馈**
- 用户可能指出建议不合理、前提错误、UI 已有该功能等
- 接收纠正，不争辩

**⑦ 自我优化**
- 用户纠正后，分析错误原因（前提没搞清？UI 没确认？安全设计没理解？）
- 将纠正点沉淀到规则文档（本节「历史教训」小节）或 memory
- 下次生成同类模块的建议时，先回顾历史教训

### 输出格式

位置：用例文档末尾，不写进用例表格。

```markdown
---

## 产品体验建议（QA 视角）

> 以下建议基于测试体验，非功能缺陷，供产品/设计参考。

### 安全风险
- 【建议】...

### 易用性
- 【建议】...

### 操作效率
- 【建议】...
```

### 历史教训（持续积累）

> 以下是过往建议中被用户纠正的错误，生成新建议前必须回顾，避免重复犯错。

| 错误类型 | 具体案例 | 正确理解 |
|---------|---------|---------|
| 前提未搞清 | 最近转账列表提了"地址投毒"风险 | 最近转账只包含用户主动发起的出账记录，攻击者发的入账不会进入该列表 |
| 建议暴露安全隐藏 | 建议观察账户/未备份钱包显示为灰色而非隐藏 | 隐藏不可用账户是安全设计，暴露反而引导用户尝试绕过限制 |
| UI 已有未确认 | 建议搜索框加 placeholder 说明搜索范围 | 搜索框已有"搜索"或"粘贴地址"的提示 |
| 过度解释用户选择 | 建议新鲜地址旁加标签解释功能 | 新鲜地址有独立开关，用户开启即理解，无需额外解释 |

## 11）数据驱动测试规范

> 当同一功能需要覆盖多组输入组合时，使用数据驱动测试方法组织用例。

### 适用场景

- 同一功能需覆盖多组输入（如金额 × Memo 组合）
- 多链/多网络测试（同一功能在不同链上的表现）
- 批量边界值验证
- 参数正交组合测试

### 用例组织方式

#### 单维度参数化

| 维度 | 取值档位 | 说明 |
|------|---------|------|
| 金额 | 最小值 / 中间值 / 最大值 | 3 档 |
| Memo | 空 / 短 / 最大长度 | 3 档 |
| 网络 | Mainnet / Testnet | 2 档 |

#### 多维度组合

| 组合方式 | 公式 | 适用场景 |
|---------|------|---------|
| 全组合 | N × M × ... | 维度少、每维取值少（≤ 9 条） |
| 正交组合 | 使用正交表 | 维度多、需要减少用例数量 |
| 关键路径 | 手动选择 | 优先覆盖高风险组合 |

**示例（金额 × Memo 组合）**：

| # | 金额 | Memo | 场景说明 |
|---|------|------|---------|
| 1 | 最小值 | 数字+字符（最大长度） | 边界组合 |
| 2 | 最大值 | 空 | 边界组合 |
| 3 | 中间值 | 纯数字 | 正常路径 |

### 精简原则

1. **优先选择关键路径组合**：非全组合，聚焦高风险场景
2. **硬件钱包用例精简**：按 `transfer-chain-rules.md` 精简为 6 条（主币 3 条 + 代币 3 条）
3. **复用边界值**：边界值在一个维度测试后，其他维度取中间值
4. **P0 用例不做全组合**：P0 仅覆盖核心路径，全组合放 P1/P2

### 用例表格格式

**数据驱动用例推荐使用以下格式**：

| 优先级 | 场景 | 输入参数 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- | --- |
| ❗️❗️P0❗️❗️ | 正常转账 | 金额=中间值<br>Memo=空 | 填写参数，提交转账 | 转账成功 |
| P1 | 边界组合-1 | 金额=最小值<br>Memo=最大长度 | 填写参数，提交转账 | 转账成功，Memo 正确显示 |
| P1 | 边界组合-2 | 金额=最大值<br>Memo=空 | 填写参数，提交转账 | 转账成功，余额清零 |

### 多链测试数据组织

| 链 | 主币 | 代币示例 | 特殊规则 |
|---|------|---------|---------|
| Stellar | XLM | USDC | Memo 必填（CEX 充值） |
| Ethereum | ETH | USDT | Gas 波动 |
| Solana | SOL | USDC | 极低 Gas |

**详细各链规则**：见 `docs/qa/rules/transfer-chain-rules.md`
